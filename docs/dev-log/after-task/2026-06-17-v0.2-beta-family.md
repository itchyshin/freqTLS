# After-task: beta (continuous-proportion) family (v0.2)

## Task goal

Add a continuous-proportion response family to profileTLS so that responses in
`(0, 1)` (e.g. snowgum PSII operating efficiency, relative chlorophyll
fluorescence) can be fitted with the same 4PL `CTmax`/`z` machinery as the count
families. Maintainer pre-approved all remaining v0.2 roadmap items ("do all of
these"); this is the first item in the suggested order, designed to bank value
while isolating the riskiest cpp refactor (covariates) for later. Design fully
worked out in `docs/dev-log/recovery-checkpoints/2026-06-17-new-session-handoff.md`.

## What was implemented

- **Engine** (`src/profile_tls.cpp`): the family branch became
  `if (family_code == 0) {binom} else if (family_code == 1) {betabinom} else
  {beta}`, the beta arm using `dbeta(y, p*phi, (1-p)*phi)` with the same shape
  floor (`1e-8`) as the beta-binomial and the existing `p` clamp; `ADREPORT(phi)`
  guard widened from `== 1` to `>= 1`. `n(i)` is an unused dummy in the beta arm.
  **Codes 0/1 are byte-identical** (the new `else if` captures exactly the old
  `else` for the only codes existing data carries).
- **Family registry** (`R/families.R`): new `beta_tls()` (`family_code = 2L`,
  `phi` with a log link); `resolve_tls_family()` now `match.arg`s
  `c("beta_binomial", "binomial", "beta")` with a `beta = beta_tls()` switch arm.
- **`fit_tls()`** (`R/fit_tls.R`): family default gains `"beta"`; `n` is now
  optional (`quo_is_missing` -> `NULL` -> dummy `rep(1, n_obs)` for beta, else a
  clear "`n` (trials) is required for the {family} family" abort); a beta
  validation branch requires `y` in `[0, 1]`, clamps exact `0`/`1` boundary
  values inward to `(1e-6, 1 - 1e-6)` with a warning, and skips the count check.
  The count families keep the identical count check (wrapped in an `else if`).
- **`simulate_tls()`** (`R/simulate.R`): family default gains `"beta"`; the beta
  arm draws `rbeta(N, p*phi, (1-p)*phi)` and returns a single `prop` column (no
  `total`/`survived`); `phi` required for beta. The binomial / beta-binomial
  draw arms are unchanged, so their fixed-seed RNG stream is byte-identical
  (verified: all pre-existing simulate-seeded tests still pass).
- **Formula interface** (`R/formula.R`): `tls_parse_response()` accepts a
  bare-name LHS as a proportion (`prop ~ time(duration) + temp(temp)`), returning
  `n = NULL`; `fit_tls()` then supplies the dummy.
- **`phi`-presence touchpoints** flipped `== 1L` -> `>= 1L` (so beta's free
  `log_phi` is recognised) in `R/utils.R` (name map), `R/profile.R` (valid-target
  message + contrast start), `R/diagnostics.R` (post-fit phi check), and
  `R/fit_tls.R` (estimate row + SE). The two binomial-only `log_phi`-map guards
  (`== 0L` in `fit_tls` and `profile.R`) were left unchanged.
- **Bootstrap** (`R/bootstrap.R`): `draw_y()` gains a beta arm (redraw `rbeta`
  proportions, clamped to `(1e-6, 1 - 1e-6)`); `phi_hat` and the `vals["phi"]`
  write widened to `>= 1`. This prevents a beta fit's bootstrap from silently
  drawing **binomial counts** (the prior `else` arm). Codes 0/1 keep the
  identical RNG path (the `fam_code == 2L` test consumes no RNG).
- **Print honesty** (`R/methods.R`): `print.profile_tls()` shows "mean observed
  proportion" for the beta family instead of the misleading "{N} survivors of {M}
  trials" (which, with the dummy `n = 1`, would have read as nonsense).

## Mathematical contract touched

A new family/likelihood was added (`docs/design/02-family-registry.md` updated):
the beta arm is `y ~ Beta(p*phi, (1-p)*phi)` with the **same `phi` = sum-of-Beta-
shapes convention** as the beta-binomial, so `phi` is comparable across the two
dispersed families. The 4PL mean map and all transforms are unchanged. The
dummy-`n = 1` design makes the count-based displays/diagnostics correct for beta
without special-casing: observed `prop = y / n = y` is exactly the proportion, so
`check_tls_data()`, `plot_survival_fit()` (`obs$y / obs$n`), and the Wald/profile
machinery (driven by the name map) all work unchanged.

## Tests of the tests

`test-fit-beta.R` confirmed RED first (9 failures, all "beta is not a known
family" / `match.arg` partial-matched `"beta"` -> `"beta_binomial"`), then GREEN
after implementation. It covers: parameter recovery (CTmax/z/phi); `n` optional
for beta and a clear error for a count family without `n`; `y` outside `[0, 1]`
aborts and exact `0`/`1` clamps with a warning; the bare-name formula response
equals the column fit; profile (CTmax and `phi`) and parametric-bootstrap
intervals are finite and ordered; grouped beta recovers distinct per-group CTmax;
and a clean beta fit is silent with a finite `phi` SE.

## Consistency audit

`family_code` semantics (0/1/2), the `phi`-presence checks (`>= 1L`), and the two
binomial-only `log_phi` map guards (`== 0L`) are aligned across `fit_tls.R`,
`profile.R`, `bootstrap.R`, `utils.R`, and `diagnostics.R`. The capability matrix,
known-limitations, ROADMAP, NEWS, README, and the family-registry design doc all
agree: beta = fitted in v0.2.

## What did not go smoothly

- `match.arg("beta", c("binomial", "beta_binomial"))` partial-matched to
  `"beta_binomial"`, so before the registry change `family = "beta"` silently
  became beta-binomial; adding `"beta"` to the choice set fixes it (the RED run
  surfaced this clearly).
- First beta validation message used cli `{?s}` pluralisation with no quantity
  (cli error); fixed by counting the offending values first (`{n_bad} value{?s}`).
- `confint()`'s bootstrap seed argument is `boot_seed`, not `seed` (corrected in
  the test).

## Known limitations and next actions

- Beta dispersion `phi` is assumed constant across the curve (no
  varying-dispersion sub-model). Genuine zero/one inflation should use the
  beta-binomial or a different tool; exact boundary values are clamped, not
  modelled.
- A flagship beta example/figure in the README and a dedicated vignette are a
  small possible follow-up (the capability is documented in prose + the matrix).
- Unrelated stale doc noticed but not changed (surgical scope): the ROADMAP
  random-effects bullet still says "ranef() ... Phase 2" although `ranef()` and
  the `sigma_CTmax` Wald interval shipped in `e01ff1e`. Flag for a docs sweep.
- Next v0.2 items (suggested order): covariate effects on `low`/`up`/`log_k`
  (the big cpp refactor; needs a `docs/dev-log/decisions.md` entry relaxing the
  midpoint-only invariant), then `T_crit`, heat-injury prediction, and shipping
  RE profiling.
