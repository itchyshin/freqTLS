# After-task: deterministic heat-injury prediction (v0.2)

## Task goal

Add `predict_heat_injury()`, the deterministic / maximum-likelihood analogue of
`bayesTLS::predict_heat_injury()`: accumulate thermal damage from the
already-fitted TLS curve under a temperature trace and read survival back off the
4PL. Fourth of the maintainer-approved ("do all") v0.2 items.

## Scope decision (recorded; the boundary question)

`docs/design/46` and AGENTS.md frame heat-injury as a `bayesTLS`-owned concern.
The maintainer was asked directly and chose **"implement (prediction only)"**.
The distinction that resolves the apparent conflict, now documented in the matrix
and known-limitations: **fitting** injury / repair dynamics (estimating them as
part of a likelihood) stays a `bayesTLS` non-goal; **deterministic prediction**
from the already-fitted profileTLS curve is the ML analogue and is in scope. This
keeps the complementary (not competitive) framing: profileTLS offers the
likelihood-path version of the same utility, and does not fit injury models.

## Model (confirmed against the bayesTLS source, adapted, not copied)

- Relative lethal time `LT(T) = tref * 10^((CTmax - T) / z)` (the curve midpoint;
  the same quantity as `derive_lt()` at the relative threshold). Damage rate
  `dmg(T) = 1 / LT(T)` (lethal doses per time unit).
- Forward Euler over the **actual per-step** increments (bayesTLS uses a single
  `dt` from the first interval and mislabels it "trapezoidal" in a comment; this
  uses real `diff(time)` and is named Euler):
  `D_j = max(0, D_{j-1} + (dmg(T_{j-1}) - rep_{j-1}) * (t_j - t_{j-1}))`, `D_1=0`.
- Survival from dose via the 4PL with the dose as equivalent `log10`-time:
  `survival(D) = low + (up - low) * plogis(-k * log10(D))`, so `D = 1` -> the
  relative midpoint survival `(low + up) / 2`. `irreversible = TRUE` makes
  survival monotone non-increasing.
- Optional damage cutoff `t_c` (e.g. from `derive_tcrit()`): `dmg = 0` at or
  below it.
- Optional Sharpe-Schoolfield `repair` (Kelvin reference temps), subtracted each
  step and scaled by the current survival fraction. The repair parameters are a
  user-supplied scenario layer, **not identified** by the survival data, so a
  warning is emitted when repair is used (profileTLS's "never silent" stance).

The `runif` rate sampling and the absolute-target dose remapping in bayesTLS were
deliberately not copied (no ML meaning / out of scope); the prediction is on the
project-default **relative** threshold.

## Checks run and exact outcomes

- `devtools::document()` -> OK (no warnings).
- `devtools::test()` -> `PASS 368 | FAIL 0 | WARN 0 | SKIP 0` (was 355; +13).
- `rcmdcheck::rcmdcheck(args = "--no-manual", env = c(_R_CHECK_FORCE_SUGGESTS_ =
  "false"))` -> `Status: OK` (0/0/0).
- Behaviour: constant 38 C exposure to `t = LT(38)` gives `dose -> 1` and
  survival -> the relative midpoint (tied to `derive_lt`); a trace below a `t_c`
  cutoff keeps `dose == 0` and survival at `up`; positive repair strictly reduces
  the final dose and raises survival.

## Tests of the tests

The headline test pins the model to `derive_lt`: one lethal time of constant
exposure must give one lethal dose and the relative-midpoint survival -- an
independent cross-check of the dose -> survival map. The cutoff, trace
validation, and repair-direction tests are deterministic; the repair test also
asserts the unidentified-parameter warning fires.

## What did not go smoothly

- `expect_warning()`'s return value is not the wrapped expression's value here, so
  the test captures the result separately under `suppressWarnings()`; the warning
  text was matched on "identified" (it reads "not identified by the data").

## Known limitations and next actions

- Relative-threshold only; absolute-target heat injury is a future extension.
- Repair is an optional, unidentified scenario layer, not estimated.
- The integrator is forward Euler; a finer trace reduces discretisation error.
- Next (final feature): covariates on `low`/`up`/`log_k` -- the engine refactor
  (scalars -> design-matrix vectors, byte-identical for the intercept-only
  default), gated by a `docs/dev-log/decisions.md` entry relaxing the
  midpoint-only invariant. Then polish (dashboard, vignettes, site).
