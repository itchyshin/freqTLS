# After-task: formula-interface grouped fits carry their grouping vector

## Task goal

Fix a pre-existing bug found while building the item-2 flagship example:
`plot_survival_curves()` errored on any formula-interface grouped fit. Maintainer
approved fixing at source (vs working around it). TDD; R-only; no engine change.

## Root cause

`fit_tls()`'s formula branch hardcoded `group_v <- NULL` ([R/fit_tls.R:106]) -- a
deliberate shortcut whose own comment said the grouping vector "is not
reconstructed". So `fit$diag_data$group` was empty (length 0) for formula-grouped
fits, while the column interface (`group =`) populated it. Consequences:

- `plot_survival_curves()` failed at the observed-point overlay
  (`obs_df$group <- as.character(obs$group)` -> 0-length into an N-row frame:
  `"replacement has 0 rows, data has N"`). The fitted curves were fine (they use
  `fit$group_levels`); only consumers of `diag_data$group` broke.
- The group-aware data-adequacy diagnostics fell back to the ungrouped checks for
  the formula path.

The bug hit every formula-grouped fit, with or without shape covariates; the
column interface and ungrouped fits were unaffected.

## Fix

For the supported single-factor grouping (`CTmax ~ group`), the design builder
already constructs the factor (`g <- factor(...)`, [R/formula.R]). It now also
returns the per-row labels (`group = as.character(g)`); `tls_parse_formula()`
surfaces them (`group = ct_design[["group"]]`); the formula branch sets
`group_v <- spec$group`. A general multi-predictor `CTmax` design has no single
grouping label, so it stays `NULL`.

Subtlety caught by a TDD guard test: the ungrouped design carries `grouped` but
no `group` key, and `ct_design$group` **partial-matches** `$grouped` (`FALSE`).
Using `[["group"]]` (exact) returns `NULL` as intended.

## Files changed

- `R/formula.R`: single-factor design returns `group`; `tls_parse_formula()`
  surfaces it via `[["group"]]` (exact match).
- `R/fit_tls.R`: formula branch `group_v <- spec$group` (was `NULL`); comment
  updated.
- `tests/testthat/test-formula.R`: +4 tests (group vector present + matches data;
  ungrouped stays `NULL`; `plot_survival_curves()` returns a ggplot).
- Docs: NEWS (bug-fix bullet), `docs/dev-log/known-limitations.md` (the stale
  "group-aware warnings fall back to ungrouped" note replaced).

## Checks run and exact outcomes

- RED: the grouped-vector test and the plot test failed (the plot with the exact
  `"replacement has 0 rows, data has 210"` error); the ungrouped-NULL guard
  passed. After a first GREEN attempt the ungrouped guard failed
  (`diag_data$group` was `FALSE` via `$` partial match) -- fixed with `[["group"]]`.
- `devtools::test(".")` -> `[ FAIL 0 | WARN 0 | SKIP 0 | PASS 402 ]` (was 398;
  +4). No regressions: the group-aware diagnostics now firing on formula fits did
  not break any existing expectation.
- `rcmdcheck::rcmdcheck(args = "--no-manual", build_args = "--no-manual",
  env = c("_R_CHECK_FORCE_SUGGESTS_" = "false"), error_on = "never")` -> recorded
  in the check-log.

## Tests of the tests

The grouped test pins `diag_data$group` to the source data column row-for-row
(`identical(as.character(...), as.character(d$group))`). The ungrouped guard pins
the `NULL` (it is what surfaced the partial-match defect). The plot test is the
behavioural regression (`plot_survival_curves()` returns a ggplot, not an error).
