# After-task: performance study fix + real comparison page (v0.2)

## Task goal

Turn the `comparing-to-bayesTLS` article into a real, reproducible head-to-head
now that the bayesTLS cache is built: correct the recipe, repair and re-run the
performance study, and rebuild the page with real numbers and the bootstrap.
Framing stays complementary/pluralistic -- two valid lenses on the same model, no
scoreboard.

## What was implemented

- `vignettes/comparing-to-bayesTLS.Rmd`:
  - Fixed the `eval = FALSE` recipe to the true bayesTLS workflow
    (`standardize_data()` -> `fit_4pl(temp_effects = "mid", family = ...)` ->
    `extract_tdt(t_ref, time_multiplier)`; `ts_stage1/ts_stage2/ts_ci`). The old
    recipe passed `t_ref`/`time_multiplier` to `fit_4pl()` (wrong) and skipped
    `standardize_data()`.
  - Replaced the bare `str()`/`print()` cache dump with a real three-way table
    (two-stage / bayesTLS / profileTLS, shrimp CTmax and z), a "how fast / how
    accurate / how calibrated" trio of tables read from the perf rds, and an
    "always an interval: the bootstrap fallback" demo.
  - Reframed "when to prefer which" around the bootstrap fallback.
- `data-raw/performance-study.R`:
  - Profile/Wald coverage and the speed profile now pass `fallback = FALSE`, so
    the perf study measures the PURE profile (the v0.2 bootstrap fallback must not
    contaminate the profile method's calibration or timing).
  - Tightened the accuracy validity filter to `code == 0 && pdHess` plus a
    plausible-CTmax band; a fit that "converges" to a boundary is an
    identifiability failure, not a recovery.

## Mathematical contract touched

None (no model/likelihood/profile-algorithm change). The matched benchmark
configuration is unchanged: beta-binomial, relative threshold, constant shape
(`temp_effects = "mid"`), `tref = 1` hour, so all three estimators target the
same fitted curve (docs/design/06-benchmark-protocol.md).

The perf study is a maintainer artifact, not unit tests; the accuracy filter was

The vignette's correctness was validated by a clean end-to-end render that reads
the committed rds via `system.file()`.

The recipe in the vignette mirrors the corrected `build_benchmark_cache.R`. The
matched-config wording (relative threshold, constant shape, tref = 1 h) is
consistent across the vignette, the cache script, and design doc 06. Compatibility
language preserved; "posterior"/"credible" appears only in the deliberate
comparison contrast.

## What did not go smoothly

- The shipped `performance_results.rds` was both stale (harder z = 2) and showed
  profile coverage 0.000 / NA width for every profile row. Diagnosis: a dead
  artifact from an interrupted/older run, not a current bug -- the current code
  gives 0.947 / 0.953. The new `fallback = TRUE` default would also have
  contaminated profile coverage, hence the explicit `fallback = FALSE` in the
  study.
- Beta-binomial "harder" accuracy blew up from boundary solutions; fixed with the
  stricter validity filter rather than by hiding the setting.
